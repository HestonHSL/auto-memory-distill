# Memory Consolidation Workflow (记忆整理工作流)

> **定位**：Layer 1 → Layer 2 的提纯器。读取 `.logs/` 的原始碎片，结合 [提纯标准](../references/distillation-standards.md) 的步骤，将其升华为 `.memory/` 中的可执行规则，并对现有规则进行去重和整理。

---

## 触发时机 (Settlement)

本工作流用于最后的规则沉淀，在以下时机触发：

- **任务结束**：当前开发任务全部完成。
- **话题转移**：AI 判定用户已进入一个全新的话题（如从前端 UI 转向数据库迁移）。

---

## 步骤详解

### 1. 读取 `.logs/` 中的碎片原材料

- 列出 `.logs/` 下所有 `.md` 话题文件。
- 逐一读取，收集尚未提纯到 `.memory/` 的条目。

### 2. 应用提纯四步法（`memory-distill` 核心逻辑）

对每一条碎片依次执行：

1.  **意图识别**：判断类型（偏好修正 / 规范落地 / 功能纠偏 / 业务补充 / 一次性）。
2.  **价值判定**：通过质量标准（可复用？可执行？可检索？不含具体路径/行号？）。
3.  **抽象成规则**：写出一句指令式语言的规则。
4.  **去重合并**：与 `.memory/` 已有条目比对，合并或覆盖冲突。

### 3. 整理现有 `.memory/` 规则

扫描所有分类文件，处理以下问题：

| 问题                                | 处理方式                           |
| :---------------------------------- | :--------------------------------- |
| 🔴 **冲突**：两条规则互相矛盾       | 保留更新的，删除旧的               |
| 🟡 **重叠**：多条规则描述同一偏好   | 合并为一条更完整的规则             |
| 🔵 **升级**：模块级规则实际全局适用 | 移入 `project-spec.md`，删除原条目 |
| ⚪ **过时**：针对旧版本/旧库的规则  | 添加 `待验证` 标记或直接删除       |

### 4. 更新分类摘要（渐进式披露）

在每个分类文件顶部写入/更新「核心规则速览」，供 AI 快速注入上下文：

```markdown
## 速览（由 /memory-consolidation 生成）

- [规则标题]: [一句话可执行规则]
- ...
  > 完整规则见下方各条目。
```

### 5. 更新全局索引 `.memory/index.md`

确保每条文件链接准确，并附 1-2 句核心规则简介，供 AI 判断是否需要读取完整文件。

### 7. 碎片结算与清理 (Cleanup)

为了防止下一次整理时重复处理相同的 Log，必须执行以下操作：

- **彻底删除**：将已成功提纯到 `.memory/` 的话题文件从 `.logs/` 中彻底删除。
