# Auto Memory Sync Workflow

> **定位**：Layer 1 写入器。轻量、快速，只负责忠实记录"发生了什么"，不要求精确分类或深度抽象。
> 需要时再通过 `/memory-consolidation` 将碎片提纯为 `.memory/` 规则。

---

## 触发时机 (Immediate Sync)

AI 分析完用户输入、代码修改或显式指令后，**立即**触发本工作流。无需等待任务结束。

- **代码修改**：用户手动修改了 AI 生成的代码。
- **显式指令**：用户明确指出的样式、逻辑或配置偏好（如：“外边距统一改成 8px”、“全部使用下划线命名”、“用 Axios 替换 Fetch”）。
- **反馈信号**：用户对方案表示否定或要求微调（如：“不这样”、“换个思路”、“这里写错了”）。
- **决策沉淀**：会话中最终确定了某个库的选型、组件结构或关键逻辑。
- **模式识别**：出现了重复的代码风格或特定的库调用约定。
- **冲突响应**：AI 发现并纠正了与旧记忆库的不符点。

---

## 步骤详解

1.  **识别意图信号**
    - 回顾本次任务：从对话中提取用户明确或隐式的偏好、否定反馈或结构化决策点（参考 [意图识别规范](../references/intent-recognition.md)）。
    - **核心原则**：不再提取手动代码改动，仅以对话中的“意图信号”为唯一记录源。

2.  **静默记录 (Automated)**
    - 默认**无需用户确认**。直接进行 Layer 1 记录。
    - 仅在极具争议或需要澄清时简单告知用户：“已将本次组件命名偏好同步至记忆日志”。

3.  **写入 `.logs/[topic].md` (Topic-Based Logging)**
    - **话题提取**：基于本次变动提炼 1-2 个精准的英文/拼音关键词作为文件名（如：`form-margin.md`, `git-commit.md`, `project-init.md`）。
    - **写/追逻辑**：若对应话题文件不存在则创建，已存在则在底部追加。
    - **按以下格式写入**：

      ```markdown
      ## [YYYY-MM-DD HH:MM] [子任务简述]

      - **具体变动**: [用户改了什么/显式指令内容]
      - **意图分析**: [偏好 / 纠错 / 约定]
      - **核心规则预览**: [提炼出的初步规则片段]
      ```

    - 一次性修改直接跳过，不写入。

4.  **实时整理 (Mandatory)**
    - 完成 Layer 1 写入后，必须立即执行 [`/memory-consolidation`](./memory-consolidation.md) 将碎片同步提纯为 Layer 2 规则。
