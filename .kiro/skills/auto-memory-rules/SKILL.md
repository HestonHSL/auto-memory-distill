# Auto-Memory-Rules 核心执行准则

> 本文件是记忆管理的"唯一真理来源"。

## 核心价值

**自动学习用户偏好，形成个性化的代码规范 checklist**

1. **自动捕获** - 在对话中识别用户的偏好和纠错，实时记录到 fragments
2. **智能提纯** - 将重复出现的偏好提炼成可执行的规则
3. **主动应用** - 在后续任务中自动遵循这些规则

## 最高优先级原则

**用户的实时要求 > 历史记忆规则**

- 当用户的当前要求与历史记忆冲突时，无条件执行用户要求
- 将冲突视为"规则更新信号"
- 在对话结束时，**询问用户**是否要覆盖旧规则
- 记忆是参考，不是约束

## 任务生命周期

### 0. 对话内缓存

在同一对话中维护以下状态以减少 IO：
- `last_keywords` - 上次的关键词列表
- `last_topic_fingerprint` - 话题指纹（前 3 个关键词）
- `loaded_rules` - 本对话已读取的规则文件
- `pending_fragments` - 待提纯的碎片列表

### 1. 对话开始（Pre-task）

1. **检查目录**
   ```bash
   ls -la .kiro/skills/auto-memory-rules/memory/ 2>/dev/null
   ```

2. **自愈初始化**（如果目录不存在）
   - 创建 `memory/` 目录结构
   - 创建 `memory/index.md` 索引文件
   - 创建 `memory/rules/` 和 `memory/fragments/` 子目录

3. **加载相关记忆规则**
   - 根据用户请求的关键词，搜索相关规则
   - 将匹配的记忆规则注入到当前上下文
   - 如果没有找到相关记忆，明确说明"未找到相关历史记忆"

### 2. 对话执行（Execution）- 核心功能

**触发条件：**

| 意图类型 | 触发信号 | 优先级 | 处理策略 |
|---------|---------|--------|---------|
| **偏好修正** | "我更喜欢..."、"应该用..." | 🔴 高 | 立即创建 fragment |
| **规范落地** | 团队/公司强制规范 | 🔴 高 | 立即创建，标注"强制性" |
| **功能纠偏** | "不对"、"应该是..." | 🟡 中 | 判定是否有模式可提取 |
| **业务补充** | 补充业务上下文 | 🟡 中 | 记录到 fragment |
| **一次性修改** | "这里特殊处理" | ⚪ 低 | **不记录**，避免污染 |

**执行动作：**
1. **立即捕获** - 创建 `memory/fragments/keyword1-keyword2-YYYY-MM-DD-HH-mm-ss.md`
2. **记录内容** - 场景、用户要求、执行结果

**重要：不需要等用户说"对了"**
- 只要用户纠错，就立即创建 fragment
- 在对话自然结束时进行结算
- 不依赖用户的明确确认

### 3. 对话结束（Settlement）

**话题切换判断：**

当出现以下任一条件，视为话题切换，触发结算：
- 明确转折词：「另外」「改成」「换成」「再来」「顺便」
- 本回合关键词与 `last_keywords` 差异 >= 40%
- 用户明确要求新增/修改规则
- 对话自然结束

**结算流程：**

1. **检查待提纯的碎片**
   ```bash
   ls -la .kiro/skills/auto-memory-rules/memory/fragments/
   ```

2. **执行记忆提纯**
   - 审查每个 fragment 的内容
   - 判断是否达到提纯标准（重复出现、明确偏好、可执行）
   - 检查是否已有相似规则（避免重复）
   - 将合格的 fragment 提纯为 rule 文件
   - 清理已提纯的 fragments

3. **更新索引和 Checklist**
   - 更新 `memory/index.md`
   - 自动生成 checklist：`.kiro/skills/auto-memory-rules/scripts/generate-checklist.sh`（输出到项目根目录 `CHECKLIST.md`）

**提纯判断标准：**
- ✅ 用户明确表达的偏好
- ✅ 重复出现的修改模式（2次以上）
- ✅ 可以抽象为通用规则
- ❌ 一次性的特殊情况
- ❌ 项目特定的实现细节

**质量门控 Checklist：**

核心问题：**这个规则下次能让用户少纠错吗？**

应该记录：
- ✅ 重复出现 2 次以上的纠错
- ✅ 明确的偏好表达（"我更喜欢..."、"应该用..."）
- ✅ 可抽象为通用规则（跨项目可用）
- ✅ 高成本的错误与修正路径
- ✅ 关键参数/设定/前置条件
- ✅ 用户偏好与风格规则

不应记录：
- ❌ 一次性的特殊处理
- ❌ 项目特定的业务逻辑
- ❌ 临时的解决方案
- ❌ 一问一答、没有可重用流程
- ❌ 纯概念解释（没有具体做法）

质量标准：
- [ ] **去具体化** - 移除文件路径、行号、具体变量名
- [ ] **高可执行性** - 规则必须是 AI 读完后能立刻执行的指令
- [ ] **跨任务通用** - 该规则在其他类似场景中依然有效
- [ ] **抗衰减** - 规则逻辑清晰，不会随时间推移而失效

**冲突处理：**
- **规则演进** - 若用户明确推翻旧习惯，询问后覆盖旧规则
- **场景分支** - 若两个规则在不同场景下均有效，则合并并详细描述各自的触发场景

## 代码审查流程

### 生成 Checklist

**自动生成时机：**
- 对话结束时，如果有新的 rule 被创建或修改

**手动生成：**
```bash
.kiro/skills/auto-memory-rules/scripts/generate-checklist.sh
```

### 触发审查

在对话中主动请求：
```
"用 checklist 审查代码"
"根据我的偏好规则检查这个文件"
```

AI 会：
1. 加载项目根目录的 `CHECKLIST.md`
2. 逐项检查代码
3. 报告所有问题
4. 询问是否自动修复

### 审查输出格式

```markdown
# 代码审查报告

## 文件: src/components/FileList.tsx

### ❌ 发现 3 个问题

#### 1. Hook 暴露了内部 setter（第 15 行）
- 规则: hook-no-expose-setter.md
- 代码: `return { fileList, setFileList };`
- 建议: 提供语义化方法如 updateFileItem
- 优先级: 🟡 中

### ✅ 符合规范（5 项）
- Hook 函数有 JSDoc 注释
- Import 顺序正确

---

是否自动修复以上问题？[Y/n]
```
